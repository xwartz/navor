# 设计哲学

本文说明 **Navor 为何这样设计**，动机见[宣言](manifesto.zh.md)，语法见[语言](language/overview.zh.md)。

## 纯文本优先

文档应该比应用活得更久，Navor 的源文件是 `.nav` 纯文本，随便什么编辑器都能打开，Git 能 diff，不跑服务也能归档。

## 人类可读

每个 `.nav` 文件都应该让人直接读懂。日期、指令、subject、元数据和 Markdown 正文先为人写，工具再去解析。

## 对 AI 友好

人读得懂的文档，AI 也用得上。观点、决策、交易这些结构化事实和叙述正文放在同一份文件里，你和 LLM 看的是同一个版本。

## 原生支持 Git

投资决策会变，所以要有版本历史。观点改了，就追加新指令，别回头改旧记录来「美化」当下。

## 规范先于实现

语言是产品，应用是实现。`.nav` 语法、`nav` CLI、Reader 输出和静态站点格式是公开接口；`@navor/*` 包用来支撑 CLI，导入 API 除非另有说明，否则不稳定。

## 小核心

语言尽量简单，复杂的事交给工具，Navor 0.1 只有十二条指令；同一个事实能用元数据说清楚，就别加新指令。

## 事实优先于命令

源文件只记录已经发生的事和你已经作出的决定，不下单、不跑策略，也不把实时报价写进源事实。

## 源事实优先于增强数据

`.nav` 里只放你确认过的事实，持仓、价值、盈亏、配置和偏离由引擎算出来。价格和汇率可以拿来增强视图，但数据缺了或过期了，不该影响读取和渲染。

## 形态参考 Beancount，不追求兼容

Navor 借鉴带日期记录、缩进元数据、posting、注释和 Git 审阅这套文本形态，但不解析 `.bean`，也不复刻 Beancount 的会计模型。`research`、`thesis`、`decision`、`review`、`journal`、`note` 是原生的投资过程事实。

## 兼容性

Navor 还没到 1.0，破坏性语法或语义变更要有变更日志、迁移说明、fixture 和诊断；1.0 起对已文档化的语言和 CLI 遵守 SemVer。
